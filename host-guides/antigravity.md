# Antigravity Host Guide

Treat Antigravity as a distinct host with strong browser, UI, and testing leverage, not as a permanent frontend-only role.

## Good Fits

- frontend implementation
- browser-backed testing
- UI regression checks
- visual or interaction evidence gathering

## Prompting Bias

- keep the exact page, route, or interaction scope explicit
- ask for reproducible evidence, not just taste judgments
- record the selected model because Antigravity may expose Gemini and Claude-family lanes
- keep pass or fail criteria concrete

## Weak Spots To Watch

- easy to drift into vague visual critique without acceptance criteria
- can become expensive if used for broad planning instead of focused evidence

## Practical Rule

Use Antigravity when the browser or UI surface is the advantage. Otherwise treat it like any other optional lane, not a required default.
